    v0.5 Code Reuse Review¶

    Time-boxed: ~30 min. Diff: 4250 lines, 12 files.

    Per-file findings¶

    1. src/arch_line_weights/poche_saas.py (NEW, 603 LOC)¶

    FALSE POSITIVE — AI24 zstd duplication. - Location: poche_saas.py:57 already imports _read_payload, _write_payload, PREFIX, CHUNK from apply_saas and reuses them in apply_saas_with_poche (lines 461, 492). The dataclass PocheSaasResult is parallel to ApplySaasResult but is intentionally distinct (poché-specific fields). Already DRY — no change needed. - Estimated savings: 0 LOC.

    Minor: compress_test_payload / decompress_test_payload / write_synthetic_test_ai (lines 515-603, ~88 LOC) live in production module but are test fixtures. They're the only test-helper code in the package. Could move to tests/conftest.py to keep the production module clean. Estimated savings: ~88 LOC out of src/, but unchanged total LOC (just relocates).

    2. src/arch_line_weights/hatch.py (+109 LOC for v0.5 materials)¶

    SHOULD FIX — repeated MultiPolygon dispatch boilerplate (16 functions, lines 374-561). - Every hatch_* function starts with the same 3-5-line preamble: 

    if isinstance(polygon, MultiPolygon):
    return functools.reduce(operator.iadd, (hatch_X(p, scale, **kw) for p in polygon.geoms), [])
    - A single @multipoly_aware decorator (or @functools.wraps-based wrapper) would replace all 16 dispatch blocks. The decorator could live next to the existing _poly_iter helper at line 75 (already there for the same purpose). Note: line 353 already uses _poly_iter correctly inside clt_layers. - Existing utility: _poly_iter at line 75 already iterates Polygon | MultiPolygon. Decorator would close the gap. - Estimated savings: ~48 LOC (16 functions × 3 lines avg).

    3. src/arch_line_weights/bridge.py (+465 LOC for backtrack + DBSCAN)¶

    FALSE POSITIVE — backtracking properly reuses existing helpers. - infer_bridges_backtrack reuses _collect_endpoints, _candidate_pairs, _direction_compatible, _crosses_existing, _polygon_count, _expected_polygon_count, _confidence (bridge.py:428-445). - _pair_endpoints is greedy and intentionally not used by _backtrack_search — backtracking needs to enumerate one pair at a time, not commit greedily. Reusing _pair_endpoints would defeat the purpose. Correct as-is. - _bridge_for_pair (lines 294-313) is a small helper that does duplicate ~5 lines from _pair_endpoints (the direction-compat + crosses-existing checks). Could extract a shared _admit_bridge(a, b, segments, min_gap) -> LineString | None predicate. - Estimated savings: ~5 LOC.

    SHOULD FIX — DBSCAN endpoint extraction duplicates _collect_endpoints indirectly. - collapse_endpoint_clusters calls _collect_endpoints (line 596) then re-extracts coords as [ep.xy for ep in endpoints] (line 600). The neighbours/centroids logic is fine; the seam is just in how endpoints feed into _dbscan. Acceptable as-is. - Estimated savings: 0 LOC (false positive on second look).

    4. src/arch_line_weights/cli.py (+~180 LOC)¶

    SHOULD FIX — --source click option duplicated 6 times (lines for inspect, apply, apply_jsx_cmd, apply_saas_cmd, poche_cmd, explain_layer). - Each command gets a 6-line @click.option("--source", ..., type=click.Choice(_SOURCE_CHOICES), default=Source.AUTO.value, show_default=True, help="...") block. - A single _source_option = click.option(...) constant or @source_option decorator would replace all 6 instances. Click idiom for this is well-established. - Estimated savings: ~30 LOC (6 × 5).

    5. src/arch_line_weights/layer_classify.py (+341 LOC)¶

    FALSE POSITIVE — RHINO_RULES vs AUTOCAD_RULES duplication is intentional. - The two rule lists look structurally similar (same (patterns, TierAssignment) shape) but the actual patterns are completely different (Rhino's ::-paths vs AIA NCS field codes like -WALL-). The dispatch table at line 308 (DISPATCH = {RHINO: ..., AUTOCAD: ...}) is the right level of indirection. Leave alone — the rules differ in content; abstracting further would obscure the per-source semantics. - Estimated savings: 0 LOC.

    6. src/arch_line_weights/presets.py (+148 LOC)¶

    FALSE POSITIVE — preset family lists are intentionally explicit. - 8 nearly-parallel Tier lists (SECTION_ISO_PRINT/SCREEN, PLAN_ISO_PRINT/SCREEN, etc., lines 129-229). Could in theory be generated by an ISOFamily builder taking (drawing-type, anchor-mm, sub-tier-shifts) but the human-written tier descriptions are documentation as much as data. Per project convention (LESSON 28: "explicit code over clever abstractions"), keep as-is. - _shift_tier_iso (line 282) is a clean shared helper, already used correctly by select_preset. - Estimated savings: 0 LOC.

    7. Test fixtures across 5 test files¶

    SHOULD FIX (small) — synthetic AI24 payload builders overlap. - tests/test_apply_saas.py:58 _build_synthetic_payload() — 25-line payload with two colored layers. - tests/test_apply_saas_poche.py:118 _build_minimal_payload(layer_names) — parametrized 18-line payload builder. - They serve overlapping purposes (synthesizing AI native PostScript fixtures with named layers). Could unify in tests/conftest.py as build_minimal_payload(layer_names, with_widths=False). - Note: the project has no conftest.py today. - Estimated savings: ~15 LOC (after merging both ~25- and ~18-line builders into one ~30-line parametrized helper, plus removing the originals).

    FALSE POSITIVE — test_bridge.py:37 _square_with_corner_gaps and _greedy_trap_fixture are bridge-specific geometry fixtures. No overlap with other test files.

    FALSE POSITIVE — test_hatch_v05.py, test_layer_classify.py, test_presets_v05.py all use pytest.parametrize over inline literal data, not shared fixtures. They're stylistically aligned but share nothing concrete to extract.

    Top 3 highest-impact consolidations¶

    1. hatch.py MultiPolygon dispatch decorator — ~48 LOC, removes 16 nearly-identical preambles, makes adding a new material a 3-line def hatch_X instead of 5-line. Highest impact on readability.

    2. cli.py --source option helper — ~30 LOC, reduces 6× copy-paste of an identical click option block. Low risk, immediately makes future per-command flag changes a one-line edit.

    3. tests/conftest.py for shared payload builders — ~15 LOC saved + cleaner test imports. Also enables moving the test-only helpers (compress_test_payload, decompress_test_payload, write_synthetic_test_ai) out of src/poche_saas.py (~88 LOC) without test breakage.

    "Leave alone"¶

    layer_classify.py RHINO_RULES vs AUTOCAD_RULES — the two rule lists look like a textbook DRY target but the patterns are semantically distinct (Rhino's CLIPPINGPLANEINTERSECTIONS vs AIA NCS -WALL-FULL). Trying to share structure would create a "configurable-rule schema" that just makes the simple dispatch harder to read. The existing DISPATCH: dict[Source, list[...]] is exactly the right depth. Per LESSONS_LEARNED §28-29 (standards-compliant defaults, explicit code) and the project's overall preference for "explicit code over clever abstractions" — keep both lists side-by-side, even though they share TierAssignment shape.

    Total estimated LOC reduction¶

    • Hatch decorator: ~48
    • CLI source option helper: ~30
    • conftest.py test fixtures: ~15
    • Bridge _admit_bridge extraction: ~5
    • Move test helpers out of src/poche_saas.py: 0 net (relocates ~88 LOC)

    Total: ~98 LOC across src/ + tests, with the biggest single win being the hatch decorator.
