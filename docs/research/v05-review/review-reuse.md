## Per-file findings

### 2. `src/arch_line_weights/hatch.py` (+109 LOC for v0.5 materials)

**SHOULD FIX — repeated `MultiPolygon` dispatch boilerplate** (16 functions, lines 374-561).
- Every `hatch_*` function starts with the same 3-5-line preamble:
```python
if isinstance(polygon, MultiPolygon):
    return functools.reduce(operator.iadd, (hatch_X(p, scale, **kw) for p in polygon.geoms), [])
```
- A single `@multipoly_aware` decorator (or `@functools.wraps`-based wrapper) would replace all 16 dispatch blocks. The decorator could live next to the existing `_poly_iter` helper at line 75 (already there for the same purpose). Note: line 353 already uses `_poly_iter` correctly inside `clt_layers`.
- Existing utility: `_poly_iter` at line 75 already iterates `Polygon | MultiPolygon`. Decorator would close the gap.
- Estimated savings: ~48 LOC (16 functions × 3 lines avg).

### 3. `src/arch_line_weights/bridge.py` (+465 LOC for backtrack + DBSCAN)

**FALSE POSITIVE — backtracking properly reuses existing helpers.**

- `_pair_endpoints` is greedy and intentionally not used by `_backtrack_search` — backtracking needs to enumerate one pair at a time, not commit greedily. Reusing `_pair_endpoints` would defeat the purpose. **Correct as-is.**
- `_bridge_for_pair` (lines 294-313) is a small helper that *does* duplicate ~5 lines from `_pair_endpoints` (the direction-compat + crosses-existing checks). Could extract a shared `_admit_bridge(a, b, segments, min_gap) -> LineString | None` predicate.
- Estimated savings: ~5 LOC.

**SHOULD FIX — DBSCAN endpoint extraction duplicates `_collect_endpoints` indirectly.**
- `collapse_endpoint_clusters` calls `_collect_endpoints` (line 596) then re-extracts coords as `[ep.xy for ep in endpoints]` (line 600). The neighbours/centroids logic is fine; the seam is just in how endpoints feed into `_dbscan`. Acceptable as-is.
- Estimated savings: 0 LOC (false positive on second look).

### 5. `src/arch_line_weights/layer_classify.py` (+341 LOC)

## Top 3 highest-impact consolidations

1. **`hatch.py` `MultiPolygon` dispatch decorator** — ~48 LOC, removes 16 nearly-identical preambles, makes adding a new material a 3-line `def hatch_X` instead of 5-line. Highest impact on readability.

2. **`cli.py` `--source` option helper** — ~30 LOC, reduces 6× copy-paste of an identical click option block. Low risk, immediately makes future per-command flag changes a one-line edit.

3. **`tests/conftest.py` for shared payload builders** — ~15 LOC saved + cleaner test imports. Also enables moving the test-only helpers (`compress_test_payload`, `decompress_test_payload`, `write_synthetic_test_ai`) out of `src/poche_saas.py` (~88 LOC) without test breakage.

## "Leave alone"

**`layer_classify.py` RHINO_RULES vs AUTOCAD_RULES** — the two rule lists look like a textbook DRY target but the patterns are semantically distinct (Rhino's `CLIPPINGPLANEINTERSECTIONS` vs AIA NCS `-WALL-FULL`). Trying to share structure would create a "configurable-rule schema" that just makes the simple dispatch harder to read. The existing `DISPATCH: dict[Source, list[...]]` is exactly the right depth. Per LESSONS_LEARNED §28-29 (standards-compliant defaults, explicit code) and the project's overall preference for "explicit code over clever abstractions" — keep both lists side-by-side, even though they share `TierAssignment` shape.
